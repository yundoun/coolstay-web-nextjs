## A. 아키텍처 — 코드 구조의 기반

> 프로젝트 규모가 커질수록 초기 아키텍처 결정의 변경 비용이 높아집니다.
> 인프라 결정(1번)과 독립적으로 논의 가능하므로 병렬로 진행할 수 있습니다.

### A-1. 디렉토리 구조 / 모듈 경계

**질문**: 도메인 기반 구조를 유지할지, 어떤 수준으로 모듈을 분리할지?

| 선택지 | 설명 | 장점 | 단점 |
|--------|------|------|------|
| **A) 현재 유지** (`domains/` 도메인별 분리) | 도메인별 components/hooks/types/data | POC와 동일, 도메인 응집도 높음 | 공유 로직 위치 모호, 도메인 간 의존성 규칙 없음 |
| **B) Feature-Sliced Design** | layer 기반 (`shared/entities/features/widgets/pages`) | 의존성 방향 명확, 확장성 | 학습 곡선, 오버엔지니어링 가능 |
| **C) 현재 + 규칙 보강** | `domains/` 유지 + 공유 레이어 명확화 + import 규칙 | 마이그레이션 비용 적음, 실용적 | 규칙 관리 필요 |

> **현재 POC 구조**:
> ```
> src/app/          ← 라우팅만
> src/components/   ← 공유 UI (layout, ui, search, accommodation)
> src/domains/      ← 도메인별 (components, hooks, types, data, api)
> src/lib/          ← API 클라이언트, 스토어, 유틸
> ```
> **현재 이슈**: 일부 도메인에 api/ 있고 일부는 없음. 공유 컴포넌트와 도메인 컴포넌트의 경계가 불명확.

### A-2. Server Component vs Client Component 경계

**질문**: React Server Components(RSC)를 어떻게 활용하는가?

| 선택지 | 설명 | 장점 | 단점 |
|--------|------|------|------|
| **A) 최소 사용** (대부분 `"use client"`) | POC와 유사, 대부분 클라이언트 | 기존 코드 변경 적음, 단순 | RSC 이점(번들 축소, 서버 데이터 접근) 못 누림 |
| **B) 적극 활용** (기본 Server, 필요 시 Client) | Next.js 권장 패턴 | 번들 크기 감소, 서버 데이터 직접 접근 | 코드 패턴 변경 큼, 팀 학습 필요 |
| **C) 점진 도입** (페이지 레벨 Server → 하위 Client) | 페이지 껍데기만 Server Component | 균형잡힌 접근 | 어디까지 Server인지 기준 필요 |

> **고려사항**: 이 결정은 2-1(렌더링 전략)과 연동됨. CSR 전체를 선택하면 RSC는 사실상 불필요.

### A-3. API 레이어 패턴

**질문**: API 호출 코드를 어떻게 구조화하는가?

| 선택지 | 설명 | 장점 | 단점 |
|--------|------|------|------|
| **A) 도메인별 분산** (현재) | `domains/home/api/`, `domains/search/api/` | 도메인 응집도 높음 | 공통 에러/인터셉터 처리 분산 |
| **B) 중앙 집중** | `src/lib/api/` 아래 도메인별 모듈 | 일관된 에러/캐싱/인터셉터 | 도메인과 거리 멀어짐 |
